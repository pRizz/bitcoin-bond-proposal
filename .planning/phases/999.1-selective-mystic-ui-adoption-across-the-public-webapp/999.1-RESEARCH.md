## Common Pitfalls

### Pitfall 1: Treating SolidJS Compatibility As Full Stack Compatibility

**What goes wrong:** Mystic's SolidJS peer range passes, so the executor assumes the package fits the app. [VERIFIED: Mystic package peer, `solid-js` local version]  
**Why it happens:** The real mismatch is Tailwind v4 CSS-first setup and dark token integration, not Solid's runtime. [VERIFIED: Mystic README; `app.config.ts`; `app.css`]  
**How to avoid:** Require all four gates: exact SHA, Tailwind v4 setup fit, dark-token fit, and dependency-surface fit. [VERIFIED: `999.1-UI-SPEC.md`]  
**Warning signs:** `withMysticUI`, `mystic-ui/tailwind/theme.css`, `.dark` setup, or lockfile churn appears in the diff. [VERIFIED: UI spec forbidden-churn scan]

### Pitfall 2: Importing A Broad Animated Package For One Static Editorial Need

**What goes wrong:** The route imports a Mystic component and pulls in particles/motion/icons even though the desired effect is a compact proof panel. [VERIFIED: Mystic package dependencies and source tree]  
**Why it happens:** Component libraries make small visuals look cheap while hiding dependency and setup costs. [ASSUMED]  
**How to avoid:** Adapt the structural idea locally and keep the dependency graph unchanged. [VERIFIED: decision matrix]  
**Warning signs:** `package.json` adds `mystic-ui`, `solid-icons`, `motion`, `solid-motionone`, `rough-notation`, `@tsparticles/*`, or `tailwindcss-animate`. [VERIFIED: Mystic package.json]

### Pitfall 3: Making The Comparison Surface More Decorative But Less Trustworthy

**What goes wrong:** A visual accent competes with status dates, review dates, source footing, and canonical state links. [VERIFIED: UI spec prohibits decorative competition]  
**Why it happens:** Mystic's component set includes attention-grabbing text, particle, beam, shimmer, and mock-device components. [VERIFIED: Mystic source tree]  
**How to avoid:** Limit visible treatment to labels, counts, source-aware context, and state links inside existing editorial panels. [VERIFIED: `999.1-UI-SPEC.md`]  
**Warning signs:** Animated text, ranking language, glow backgrounds, or hidden source/freshness copy. [VERIFIED: UI spec disallowed enhancements]

### Pitfall 4: Skipping Browser QA Because The Change Is "Just Layout"

**What goes wrong:** Long state names, bill IDs, or wrapped action links overflow on mobile even when TypeScript and unit tests pass. [VERIFIED: `999.1-UI-SPEC.md`; Phase 17 browser QA route/viewport evidence]  
**Why it happens:** The existing tests are model-focused and do not prove mobile wrapping. [VERIFIED: `content.test.ts`; `rg --files` test scan]  
**How to avoid:** Run browser QA at 390x844, 768x1024, and 1280x900 for `/states/compare`, with click-through and overflow checks. [VERIFIED: UI spec]  
**Warning signs:** `scrollWidth > viewportWidth`, clipped chips/CTAs, or focus rings hidden against the dark canvas. [VERIFIED: Phase 17 browser QA checks]
