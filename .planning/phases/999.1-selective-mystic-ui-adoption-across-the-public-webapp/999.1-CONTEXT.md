# Phase 999.1: Selective mystic-ui adoption across the public webapp - Context

**Gathered:** 2026-06-03
**Status:** Ready for planning
**Mode:** Yolo

<domain>

## Phase Boundary

Phase 999.1 evaluates whether `mystic-ui` should be selectively adopted on the public SolidStart webapp after the v1.3 registry expansion. It may ship a narrow proof-surface or comparison-surface enhancement only if the current Mystic UI package path proves compatible with this repo's Tailwind CSS v4 setup, dark editorial design system, trust cues, performance expectations, and existing route structure. It does not own a wholesale visual redesign, a global chrome rewrite, new state publication, database/admin changes, live-tracker behavior, or unproven dependency churn.

</domain>

<decisions>

## Implementation Decisions

### Adoption gate

- **D-01:** Treat Mystic UI adoption as evidence-gated, not automatic. The current `pRizz/mystic-ui` README describes the GitHub source package as a Vite + SolidJS + Tailwind CSS 3.x consumer path, while this repo runs Tailwind CSS v4.
- **D-02:** If a Mystic UI dependency is adopted, it must be pinned to the exact current GitHub commit SHA and verified locally. Do not use the stale npm `mystic-ui` package or a floating GitHub branch ref.
- **D-03:** If Tailwind v4 compatibility requires broad config rewrites, class-based dark-mode changes, new global token imports that override existing editorial CSS, or fragile source-shipped dependency workarounds, record a no-adoption decision and close the phase with evidence.

### Product surface

- **D-04:** Prefer `/states/compare` and proof-oriented registry surfaces as the first candidate areas because they map directly to UI-01 and SITE-13. `/states/clusters` may be considered only if the comparison/proof need is clearer there.
- **D-05:** Preserve the existing dark editorial token system, route hierarchy, source-trail posture, `StateCard`, `RegistryFreshnessPanel`, `ActionLink`, and `PageSection` patterns unless a small Mystic UI-derived treatment materially improves comprehension.
- **D-06:** Do not add a matrix, advanced filter, ranking table, or search surface unless the plan first proves the current seventeen-record registry exceeds the existing editorial route patterns.

### Implementation scope

- **D-07:** Favor a small local adaptation or theme enhancement over importing a large animated component if that delivers the same reader value with less dependency and compatibility risk.
- **D-08:** Any visible enhancement must keep provenance, status dates, freshness cues, canonical state-detail links, and comparison context more legible, not merely more decorative.
- **D-09:** Avoid changes to shared site chrome, footer, metadata, README, profile, or identity surfaces unless route-level implementation evidence makes them necessary. If those surfaces become necessary, apply the OpenLinks identity-presence guidance before editing them.

### Verification

- **D-10:** Verification must include dependency/compatibility checks, TypeScript, build, tests, precommit, and targeted browser checks on changed routes at mobile, tablet, and desktop widths.
- **D-11:** Browser QA must confirm no horizontal overflow, clipped controls, overlapping text, broken links, missing source-trail copy, or motion/accessibility regressions on every changed route.
- **D-12:** The final phase evidence must state whether Mystic UI was adopted, partially adapted, or rejected for now, and why.

### the agent's Discretion

- Decide whether the strongest result is a dependency adoption, a local visual enhancement inspired by the evaluated component set, or a documented no-adoption outcome.
- Select the narrowest public route surface that demonstrates material editorial value without changing unrelated pages.
- Add focused tests where pure route model behavior changes; otherwise rely on existing route tests plus browser/build verification.

</decisions>
