# Requirements: Bitcoin Bond Proposal

**Defined:** 2026-04-09  
**Milestone:** v1.2 Registry Expansion and Refresh Workflows  
**Core Value:** Produce a credible, reusable Illinois-first legislative package and research registry that makes Bitcoin reserve and bond policy understandable, sourceable, and practical enough for real state-level adoption.

## v1.2 Requirements

### Registry Coverage

- [ ] **CATA-07**: Reader can access a materially broader national batch of publishable state entries beyond the initial five-state set.
- [ ] **CATA-08**: Reader can see consistent summaries, source links, `status as of`, and `last reviewed` metadata across newly published state entries.

### Refresh Workflow

- [ ] **PIPE-04**: Maintainer can identify which published state entries are due for review from repo-owned freshness signals.
- [ ] **PIPE-05**: Maintainer can run a repo-owned refresh workflow or report that checks manifest state, published entries, and freshness metadata without implying real-time tracking.
- [ ] **PIPE-06**: Maintainer can update or add a state entry while validation and compile tooling enforce manifest boundary and required freshness fields.

### Public Coverage Surfaces

- [ ] **SITE-07**: Reader can browse expanded registry coverage with clear distinctions between published, queued, and unresearched states in public catalog surfaces.
- [ ] **SITE-08**: Reader can open refreshed state detail pages that make source-first trust cues and review recency obvious.
- [ ] **SITE-09**: Reader sees selective `mystic-ui` enhancements on chosen public surfaces only where they strengthen hierarchy or motion without reducing readability or trust.

## Future Requirements

### Public Comparisons

- **SITE-10**: Reader can compare states through dedicated cluster pages by region, proposal type, or legislative status.
- **SITE-11**: Reader can access richer infographic or comparison surfaces that explain reserve-versus-bond policy patterns.

### Theme Options

- **THEME-06**: Reader can choose between curated visual themes without fragmenting the editorial identity.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Real-time legislative tracking | Snapshot-based review remains more credible than implied live status |
| Full `mystic-ui` rewrite of the webapp | The milestone only allows selective compatibility-checked adoption |
| Theme toggle implementation | Coverage and refresh workflows are higher priority than another visual-only pass |
| Database-backed admin tooling | The content-first repository model remains sufficient for this milestone |
| Municipal or non-state proposal coverage | State-level breadth still matters more than scope expansion |
