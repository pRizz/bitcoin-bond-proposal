# Methodology Memo

## Purpose

Bitcoin Bond Proposal is a curated policy and research product, not a live legislative tracker and not a legal filing service. This memo explains how the project classifies proposals, what evidence is required before publication, and how readers should interpret legislative status, summaries, and draft language.

## Editorial Stance

The project is explicitly favorable to state Bitcoin reserve and Bitcoin-backed bond policy, but it does not treat advocacy as a substitute for sourcing. The site and packet are intended to be assertive, professional, and auditable. Facts, interpretation, and policy argument should remain distinguishable even when they appear in the same document.

## Source Hierarchy

### Primary sources

Primary sources are mandatory for every publishable state entry:

- official state legislature bill pages;
- official bill or enrolled-law text;
- official committee materials or fiscal notes when available;
- official state press releases or agency notices directly tied to the proposal.

### Secondary sources

Secondary sources may provide context, but they do not replace primary-source support:

- reputable reporting;
- legal commentary;
- policy analysis.

### Tertiary sources

Tertiary sources may help identify leads, but they are not sufficient authority for publication:

- advocacy summaries;
- social posts;
- unsourced trackers.

## Publishability Threshold

A canonical public state entry is publishable only when it includes all of the following:

- an official bill or law identifier;
- current legislative status;
- `Status as of` date;
- `Last reviewed` date;
- one or more primary source links;
- a plain-English summary;
- a “what this would actually do” explanation;
- reserve-versus-bond classification;
- a confidence or completeness note.

Incomplete work should remain outside the canonical public set until this threshold is met.

## Classification Rules

### Top-level proposal kind

Every proposal must be classified as one of:

- `reserve`
- `bond`
- `both`

### Proposal subtype

The initial controlled subtype set is:

- `strategic-reserve`
- `reserve-financing-bond`
- `bitcoin-linked-bond`
- `bitcoin-collateralized-bond`
- `mixed-digital-asset-reserve`
- `other-explained`

If a proposal is ambiguous, broad, or mixed, the entry must carry a short editorial note explaining why the chosen subtype is still defensible.

## Freshness Rules

This project is a curated snapshot, not a live legislative feed.

Every published state entry must display both:

- `Status as of [date]`
- `Last reviewed [date]`

Those two fields serve different purposes:

- `Status as of` identifies the date of the legislative status claim.
- `Last reviewed` identifies the last date the project re-checked the entry.

If either date is missing, the entry is not publishable.

## Caution Language

- This project is policy analysis only, not legal or investment advice.
- Draft bill language is illustrative and should be reviewed by licensed counsel and legislative drafting staff before formal introduction.
- Legislative status can change quickly.
- Official bill text controls over summaries.
- Bond structures may require additional constitutional, budgetary, tax, and securities analysis that varies by state.

## Treatment of Broad Digital-Asset Proposals

Broad digital-asset proposals belong in the canonical public set only when their Bitcoin relevance is explicit enough to defend in plain English. If the Bitcoin connection is too indirect or speculative, the proposal should remain a lead or deferred research item instead of being published as a core entry.

## Expected Public Wording

Where public entries present legislative status or legal effect, the preferred pattern is:

1. plain-English summary;
2. what this would actually do;
3. `Status as of`;
4. `Last reviewed`;
5. primary source links;
6. confidence note where needed.

This order keeps the page usable for ordinary readers without weakening the source trail.
